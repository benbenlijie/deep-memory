# deep memory：一艘不会失忆的 AI 科研船

如果你退后一步看，今天很多 AI Agent 的问题并不是“不够聪明”。

它们已经可以写代码、查资料、调用工具、跑测试、拆任务，甚至能在一个复杂项目里连续工作很久。

但它们仍然有一个很基础的问题：**记不住。**

不是说它完全没有上下文，而是说它缺少一种稳定的长期记忆能力：

- 这次任务里发现的关键事实，下次还能不能用？
- 一个 Agent 踩过的坑，另一个 Agent 能不能知道？
- 用户偏好、项目约束、架构决策，能不能跨会话延续？
- 过去的测试证据、失败路径、决策依据，能不能被未来的 Agent 检索到？

这就是 deep memory 想解决的问题。

它不是简单的“聊天记录总结”，也不是把所有历史对话都塞进向量库。更准确地说，它想成为一层 **面向 AI Agent 的长期记忆基础设施**。

我用一个故事解释。

---

## 一支聪明但健忘的科研船队

想象你有一支 AI 科研船队。

船队里有很多船员，也就是很多 Agent：

- 一个 Agent 负责写代码
- 一个 Agent 负责查资料
- 一个 Agent 负责跑测试
- 一个 Agent 负责写文档
- 一个 Agent 负责维护 Kanban 和项目状态
- 还有一个 Agent 负责复盘和总结经验

这些 Agent 单个都很聪明。

问题是，它们像一群**聪明但健忘的船员**。

今天，代码 Agent 出海探路，发现一个重要事实：

> 这个港口北侧有暗礁，大船不能直接进去，要从东侧航道绕。

如果没有长期记忆，这个发现只存在于今天这次对话里。

第二天，换了文档 Agent 上船。它不知道北侧有暗礁，于是继续按旧路线写说明。

第三天，测试 Agent 来了。它也不知道之前发生过什么，又从北侧进入，船再次搁浅。

于是你会发现一个很荒谬的现象：

> 每个 Agent 都很聪明，但整个系统没有复利。

单次任务表现很好，长期协作却像一直在原地重启。

这就是很多 Agent 系统今天的真实状态。

---

## deep memory 像一个共享地图室

deep memory 想做的事，可以想象成给这支船队建一个共享地图室。

每个 Agent 出海后，如果发现了真正重要的东西，不是只在当前对话里说一遍，而是要把它写进航海日志：

```text
发现：北侧航道有暗礁
证据：某次测试日志 / 某次执行结果
影响：以后不要从北侧进入
适用范围：这个项目、这个环境、这个版本
置信度：高
```

下一次，无论是同一个 Agent 回来，还是另一个 Agent 接手，它都可以先进入地图室：

```text
我现在要处理 deep-memory 的 contributor onboarding。
过去有没有相关经验？
有哪些坑？
哪些结论已经验证过？
哪些事情只是猜想？
```

地图室不会把所有历史记录都搬出来。

它会根据当前任务，找出最相关、最新、最可信的几条记忆，交给 Agent。

于是 Agent 不再从零开始。

这就是 deep memory 的核心：

> 把短期对话中的重要经验，转化成长期、结构化、可检索、可共享、可修正的记忆层。

---

## 新记忆是怎么产生的？

真正有趣的问题是：deep memory 怎么知道什么该记？

它不应该把所有东西都记下来。

比如：

```text
“我刚刚喝了杯水。”
```

这通常不值得进入长期记忆。

但下面这些信息就很重要：

```text
用户偏好：中文为主，技术术语用英文
项目目标：deep-memory 希望成为 Agent 长期记忆层
工程事实：fresh clone 下测试曾出现 import path 问题
架构决策：GitHub 承载开源协作，Kanban 承载 Agent 执行审计
方法论：复杂任务用 Loop Engineering 推进
```

所以 deep memory 的第一步不是“存储”，而是 **判断价值**。

一个信息要不要被记住，至少要看几个问题：

1. 它以后还会影响决策吗？
2. 它是长期事实，还是临时状态？
3. 它有证据吗，还是只是推测？
4. 它属于用户偏好、项目知识、任务进展，还是可复用方法？
5. 它会不会很快过期？

如果答案表明它值得保留，Agent 才会把它抽取成一条结构化记忆。

这个过程大概是：

```text
事件发生
  ↓
Agent 观察到关键信息
  ↓
判断是否值得长期保存
  ↓
抽取成结构化事实
  ↓
标注来源、范围、时间、置信度
  ↓
写入 memory store
```

这件事的关键 bottleneck 不在“能不能存”，而在“能不能存对”。

---

## 记忆不是一段模糊摘要，而是一张结构化卡片

普通聊天总结可能会写成：

```text
今天测试有点问题。
```

这对未来 Agent 其实帮助不大。

因为它不知道：

- 是什么测试？
- 在什么环境失败？
- 失败证据是什么？
- 是否已经修复？
- 对未来任务有什么影响？

deep memory 更理想的方式，是把记忆写成一张结构化卡片：

```yaml
type: project_fact
project: deep-memory
content: fresh clone 环境下 pytest collection 曾出现 ModuleNotFoundError
scope: contributor_onboarding
impact: 影响外部开发者首次运行体验
evidence: session / log / test output
confidence: high
created_at: 2026-06-27
updated_at: 2026-06-27
status: active
```

为什么要这么麻烦？

因为 Agent 未来用记忆时，不只是要读到一句话。它还要知道这句话的边界：

- 这条记忆属于哪个项目？
- 它是用户偏好，还是工程事实？
- 它是已经验证的结论，还是未验证假设？
- 它有没有过期？
- 它会影响哪个决策？
- 它的证据在哪里？

如果没有这些结构，memory 很容易变成一堆看似有用、实际危险的摘要。

因为旧信息会误导，新信息会被淹没，推断会被误当事实。

---

## 跨会话记忆是怎么发生的？

现在假设今天的对话结束了。

明天你开启一个新会话，问：

> deep-memory 下一步该做什么？

普通 Agent 会像第一次见到这个项目一样，从零开始分析。

但有 deep memory 时，系统会先做一件事：

```text
根据当前用户、项目和问题，
从长期 memory 中检索相关记忆，
注入到 Agent 的上下文里。
```

这就像新船员上船前，先去地图室看几页关键航海日志：

```text
这艘船的目标是什么？
之前在哪里遇到过暗礁？
船长有什么偏好？
当前任务有哪些未解决 blocker？
哪些路线已经试过？
```

于是新会话里的 Agent 不是从零开始，而是带着已有背景进入任务。

跨会话记忆可以简化成：

```text
Session A 中发现重要事实
  ↓
写入长期 memory
  ↓
Session B 开始
  ↓
检索相关 memory
  ↓
注入当前上下文
  ↓
Agent 基于这些记忆继续推理和行动
```

这就是为什么 memory 会影响 Agent 的回答。

它不是神秘地“学会了”。

更准确地说，是过去的关键事实被保存下来，并在未来合适的时候重新进入工作上下文。

## 跨 Agent 记忆是怎么发生的？

跨 Agent 记忆也类似，只不过读取者换了。

假设代码 Agent 今天发现：

```text
fresh clone 下 pytest 会因为 import path 失败。
```

它把这件事写入 project memory。

第二天，文档 Agent 要写 contributor onboarding 文档。

它检索相关记忆时，会读到：

```text
fresh clone 曾出现 import path 问题；这会影响外部开发者首次运行体验。
```

于是文档 Agent 会自然补充安装说明、测试命令和常见故障排查。

第三天，Kanban Agent 要拆 launch readiness 任务。

它也可以检索到同一条记忆，于是创建或更新任务卡：

```text
修复 fresh clone 可运行性，是 launch 前 gate。
```

第四天，Review Agent 审查 PR。

它读到历史决策：

```text
不要把 retrieval/core、packaging/test harness、lockfile churn 混在一个 PR 里。
```

于是它会要求拆分 review 边界。

这就是跨 Agent memory：

```text
Agent A 写入 memory
Agent B 检索 memory
Agent C 基于 memory 做计划
Agent D 基于 memory 做审查
```

它们不是靠“同一个大脑”共享记忆，而是通过一个共同的 memory layer 建立一致的项目世界模型。

这件事对长期协作非常重要。

否则多个 Agent 会各自维护一套局部世界观，最后互相冲突。

## memory retrieval：地图室管理员怎么找资料？

还有一个关键问题：memory store 里可能有很多记忆，不能全部塞给 Agent。

所以需要检索和排序。

你可以把它想象成地图室管理员。

Agent 进来说：

> 我要处理 deep-memory 的 launch readiness。

管理员不会把所有航海日志都搬出来，而是找出最相关的几类信息：

```text
项目目标：100k stars
当前阶段：pre-launch
关键 blocker：fresh clone、benchmark、community onboarding
用户偏好：中文为主，结构化证据
方法论：Loop Engineering + Kanban
```

这个过程通常会结合几种信号：

1. **语义相关性**
   当前问题和哪些记忆意思最接近？

2. **实体匹配**
   是否涉及同一个用户、项目、repo、任务、Agent？

3. **时间新鲜度**
   旧记忆是否仍然有效？

4. **类型优先级**
   用户偏好、项目约束、架构决策通常比普通对话片段更重要。

5. **证据强度**
   有日志、测试、commit、Kanban 记录支撑的记忆更可信。

6. **适用范围**
   一条 memory 是只适用于某个项目，还是适用于所有任务？

所以 Agent 获取记忆的过程不是简单关键词搜索，而更像：

```text
当前任务 query
  ↓
召回候选 memory
  ↓
按相关性 / 新鲜度 / 置信度 / 类型排序
  ↓
过滤掉过期或不适用的记忆
  ↓
只把最关键的几条注入上下文
  ↓
Agent 基于这些记忆推理
```

这就是 deep memory 真正影响 Agent 的地方。

## 有记忆的 Agent 和没记忆的 Agent，有什么区别？

没有 deep memory 的 Agent 像临时船员：

> “告诉我今天要干什么？”

有 deep memory 的 Agent 更像老船员：

> “我知道我们为什么出海；我知道之前哪条航线失败过；我知道船长偏好怎么决策；我知道哪些问题还没验证；我知道下一步最该检查哪个仪表。”

具体来说，它会带来五种变化。

### 1. 少重复解释

你不用每次都说：

```text
我们这个项目叫 deep-memory。
目标是做 Agent 长期记忆层。
回答用中文。
复杂任务用 Loop Engineering。
Kanban 用来承载执行审计。
```

这些都可以成为长期记忆。

### 2. 少重复犯错

如果之前发现某条路径失败过，Agent 下次不应该再盲目重复。

它应该先检查历史证据，或者直接避开已知坑。

### 3. 决策更连续

如果之前已经判断：

```text
当前 bottleneck 不是继续堆新功能，而是 launch readiness 和 contributor onboarding。
```

那下一次讨论 roadmap 时，Agent 不应该又回到“先加十个 feature”。

memory 会让决策路径更连续。

### 4. 多 Agent 协作更一致

代码 Agent、文档 Agent、Kanban Agent、Review Agent 都围绕同一套项目事实工作。

这会减少很多协作噪音：

```text
一个 Agent 以为问题是 A，
另一个 Agent 以为问题是 B，
第三个 Agent 又重新规划 C。
```

### 5. 形成复利

每完成一轮任务，系统都会多一点经验：

```text
哪些路径有效
哪些路径失败
哪些测试最能证明正确性
哪些文档容易误导用户
哪些约束不能违反
```

长期看，这就是 compound progress。

Agent 不再只是一次性工具，而更像一个持续学习的工作系统。

---

## 但 deep memory 不能只是“自动记”

这里有一个很重要的边界。

如果 memory 系统只是自动保存所有内容，它很快会变成负资产。

因为记忆会：

- 过期
- 冲突
- 重复
- 被错误推断污染
- 把临时状态误认为长期事实
- 把没有证据的猜测当作确定结论

比如一条旧记忆说：

```text
fresh clone 下测试失败。
```

后来问题修好了。

如果 memory 不更新，未来 Agent 仍然会被旧信息误导。

所以完整的 memory lifecycle 应该是：

```text
Observe 观察
  ↓
Extract 抽取
  ↓
Classify 分类
  ↓
Store 存储
  ↓
Retrieve 检索
  ↓
Use 使用
  ↓
Verify 验证
  ↓
Update / Forget 更新或遗忘
```

真正关键的不是“让 Agent 永远记住”，而是：

> 让 Agent 记得准确、记得有边界、记得可追溯，并且在记忆过期时能够修正或遗忘。

这才是 deep memory 和普通历史记录最大的区别。

---

## 最后总结

deep memory 像给一支 AI 船队建了一个共享地图室。

每个 Agent 出海时，都会把真正重要的发现写入航海日志；下一次，无论是同一个 Agent 回来，还是另一个 Agent 接手，都可以先读地图、查暗礁、看历史航线，再决定怎么走。

所以它做到的不是简单“存聊天记录”，而是：

> 把短期任务中的重要经验，转化成长期、结构化、可检索、可共享、可修正的记忆层。

如果这个 memory layer 做对了，Agent 就不再只是一个聪明的临时助手。

它会逐渐变成一个长期协作者：知道你是谁，知道项目为什么这样设计，知道哪些路径试过、为什么失败，也知道下一轮应该从哪里开始。

这也许就是 AI Agent 从“单次能力很强”走向“长期系统真正有用”的关键一步。
